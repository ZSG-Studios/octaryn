Octaryn Architecture

Owner boundaries and contracts.

Octaryn is organized around ownership. Each owner has a focused responsibility, clear inputs and outputs, and a narrow API boundary.

Owners

Owner Responsibility Examples
octaryn-client Presentation and local client host behavior. Windowing, input, rendering, shaders, overlays, client snapshots, local prediction.
octaryn-server Authority and server host behavior. Simulation, validation, persistence, server ticks, replication, command handling.
octaryn-shared Contracts that multiple owners can safely reference. IDs, snapshots, manifests, capability IDs, host API IDs, scheduling contracts.
octaryn-basegame Bundled gameplay and content module. Blocks, items, materials, recipes, tags, interactions, content data, game rules.
tools Repository-wide developer operations. Build wrappers, validation probes, profiling helpers, workspace UI.
cmake Build policy and target construction. Owner targets, dependency aliases, platform facts, toolchains.

Contract Flow

Shared contracts describe what can cross owner boundaries. Implementation remains inside the owning module.

  • Client and server share IDs, commands, snapshots, and value types through octaryn-shared.
  • Basegame uses shared module APIs and does not reach into client or server internals.
  • Client rendering consumes presentation data and snapshots; it does not own authority.
  • Server authority accepts commands, validates them, persists state, and emits snapshots.

Module Policy

Module access is explicit and validated. A module manifest requests capabilities, host APIs, package access, framework API groups, content declarations, assets, dependencies, and scheduled systems.

  • Capabilities define what a module is allowed to do.
  • Host API IDs define what host surfaces it can read or write.
  • Scheduled systems declare read/write resources and ordering.
  • Content and asset IDs must be module-owned.

Texture Pack Flow

Texture packs are content input, not shared implementation. Basegame owns pack import and atlas decisions, shared owns only the asset declarations, and client owns rendering the generated atlas.

Step Owner
Pack import rules and atlas layer names octaryn-basegame/Tools/
Block face references and content IDs octaryn-basegame/Data/
Manifest asset contracts octaryn-shared/
GPU upload, sampling, shaders, and presentation octaryn-client/

Runtime Ownership

The host owns threads, scheduling, activation, validation, and barriers. Modules supply declarations and logic; they do not create private execution systems.

Concern Owner
Frame presentation Client
World authority Server
Gameplay rules Basegame or another module
Cross-owner contracts Shared
Build orchestration Tools and CMake
