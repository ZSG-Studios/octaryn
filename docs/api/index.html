Octaryn API

Contracts first. Owners stay separate.

The Octaryn API is the narrow, implementation-free boundary between client, server, basegame, and future modules. It exposes stable contracts, IDs, manifest policy, scheduling declarations, world value types, commands, and snapshots without leaking host internals.

The API is intentionally strict and grows only through explicit owner and capability boundaries.

Owner Model

Owner Owns Must Not Own
octaryn-client Windowing, input, rendering, shaders, overlays, prediction, client host integration. Server authority, persistence, save ownership, authoritative validation.
octaryn-server Simulation, validation, persistence, server ticks, replication, transport hosting. Presentation rendering, UI, GPU upload, client-only prediction.
octaryn-shared Contracts, IDs, manifests, capability policy, scheduling contracts, value types. Client rendering implementation, server persistence implementation, product gameplay rules.
octaryn-basegame Bundled gameplay rules, blocks, items, recipes, tags, interactions, content data. Host internals, transport implementation, persistence internals, rendering internals.

Shared Contracts

octaryn-shared/ is the API surface. It stays package-light, implementation-free, and stable enough for client, server, basegame, and future modules to agree on behavior.

Namespace Purpose
Octaryn.Shared.GameModules Module manifests, registration, instances, validation reports, and frame contexts.
Octaryn.Shared.ApiExposure Capability IDs and host API IDs exposed to modules.
Octaryn.Shared.FrameworkAllowlist Approved package and framework API groups.
Octaryn.Shared.Host Scheduling declarations, resource access, frame snapshots, command sinks, and host work policy.
Octaryn.Shared.Networking Client command frames, server snapshot headers, replication IDs, and replication changes.
Octaryn.Shared.World Block IDs, block positions, chunk constants, chunk positions, and chunk snapshots.

Module Registration

Modules expose a registration object with a manifest and an instance factory. Basegame uses the same path future game modules and mods should use.

public interface IGameModuleRegistration
{
    GameModuleManifest Manifest { get; }

    IGameModuleInstance CreateInstance(ModuleHostContext context);
}

Manifest Fields

GameModuleManifest declares identity, compatibility, requested capabilities, host APIs, runtime packages, build packages, framework API groups, dependencies, content, assets, scheduled systems, and multiplayer/save compatibility.

Field Group Purpose
Identity ModuleId, display name, semantic version, and Octaryn API version.
Capabilities Exact permission request for content, gameplay, command, or world access.
Host APIs Specific host surfaces the module can read or write, such as frame data or command sinks.
Allowlists Runtime packages, build packages, and framework API groups allowed for module code.
Declarations Content, assets, dependencies, and scheduled systems owned by the module.

Deny-By-Default Validation

GameModuleValidator validates manifests before activation. A package version pin does not grant module permission; every capability, host API, runtime package, build package, and framework API group must be explicitly allowed.

  • Identity and compatibility fields are required.
  • Versions and host API ranges must be valid.
  • Capabilities, dependencies, content, assets, and scheduled systems must be unique.
  • Content, asset, and system IDs must be module-owned.
  • Schedule graphs and resource access conflicts are checked.
  • Command writes require command host API access and the matching capability.

Scheduling

Modules declare work. Hosts own execution. Scheduled systems declare phase, owner, read/write resources, ordering, flags, and commit barrier.

public sealed record ScheduledSystemDeclaration(
    string SystemId,
    HostWorkPhase Phase,
    string FrameOrTickOwner,
    IReadOnlyList<ScheduledResourceAccess> Reads,
    IReadOnlyList<ScheduledResourceAccess> Writes,
    IReadOnlyList<string> RunsAfter,
    IReadOnlyList<string> RunsBefore,
    HostWorkScheduleFlags Flags,
    string CommitBarrier);

Commands And Snapshots

Shared networking contracts define message shape, not transport implementation. Client and server owners decide how bytes move.

Contract Role
ClientCommandFrame Versioned command batch for a tick.
ServerSnapshotHeader Versioned snapshot header with replication and change counts.
ReplicationChange Compact change payload shared by native and managed owners.
ReplicationId Stable replicated-state identity.

Naming

Names should make ownership obvious. Public API names use the exact concept they expose: command, snapshot, registry, query, host, tick, schedule, or system. Avoid broad buckets and unclear file names.

  • Use owner names in public host boundaries when the owner matters.
  • Use stable module IDs such as octaryn.basegame or example.demo.
  • Keep contract files focused on one API concept.
  • Do not expose client, server, or native implementation namespaces to modules.

Module Security

Modules are deny-by-default. Every dependency, framework API group, host API, capability, content declaration, asset declaration, and scheduled write is validated before activation.

  • No direct filesystem, process, reflection, networking, native interop, or raw threading access without an explicit future contract.
  • No private worker pools, custom schedulers, timers, or unmanaged worker loops.
  • No implicit access to rendering, persistence, transport, or bridge internals.
  • Diagnostics should go through approved host diagnostics APIs as those are exposed.

Threading

The host owns all threading: one main thread, one coordinator thread, and a worker pool. Modules must not create raw threads, private schedulers, timers, unmanaged worker loops, or private worker pools.
